# Feature: Target Role Benchmark Engine

> [!IMPORTANT]
> **⚠ AI-Generated Suggestions — Human Review Required**
>
> Benchmarks produced by this engine are generated by a language model.
> They must be reviewed and validated by a **human recruiter or hiring manager**
> before being used in any hiring decision, job posting, or candidate evaluation.
> The model may hallucinate, mis-classify, or omit information even when
> structured-output validation passes.
> The API response always includes `"review_required": true` to signal this to
> downstream consumers.

---

## Overview

The **Target Role Benchmark Engine** is the core AI pipeline of the Career Coach
backend. It accepts a raw, unstructured job description string and returns a fully
structured, validated `RoleBenchmark` object — together with a 1536-dimensional
pgvector embedding — ready for semantic search and gap analysis.

---

## Architecture

### LangGraph State Machine

The pipeline is implemented as a compiled **LangGraph `StateGraph`**. Each execution
is a pure function over a typed `BenchmarkState` dictionary, making the flow
deterministic, inspectable, and easy to test.

```
START
  │
  ▼
┌────────────────────────────────────┐
│  [ extract ]                       │
│  ChatOpenAI.with_structured_output │
│  → RoleBenchmark (Pydantic v2)     │
└──────────────┬─────────────────────┘
               │
    ┌──────────▼──────────┐
    │  route_after_extract │
    └──────────┬──────────┘
               │
   ┌───────────┼───────────────┐
   │           │               │
success    retry (< 3)    exhausted (= 3)
   │           │               │
   ▼           ▼               ▼
[ embed ]  [ extract ]    ValueError
   │         (loop)
   ▼
[ save_benchmark() ]
   │
  END
```

#### Nodes

| Node | Responsibility |
|---|---|
| `extract` | Calls the LLM with a structured-output chain. On `ValidationError`, increments `error_count` and records the Pydantic error messages. |
| `embed` | Joins all skills and tools into a single string corpus and calls `OpenAIEmbeddings.embed_query()` to produce a 1536-dim vector. |

#### Service Layer

| Function | Responsibility |
|---|---|
| `run_benchmark_pipeline()` | Executes the full extract + embed LangGraph pipeline and returns the final `BenchmarkState`. |
| `save_benchmark()` | Maps the `BenchmarkState` into a `RoleBenchmarkModel` ORM record and commits it to PostgreSQL. |

#### State: `BenchmarkState`

| Key | Type | Description |
|---|---|---|
| `raw_text` | `str` | Original job description (immutable throughout the run) |
| `extracted_data` | `RoleBenchmark \| None` | Populated after a successful extraction |
| `embedding` | `List[float] \| None` | 1536-dim vector, populated after the embed node |
| `error_count` | `int` | Number of failed LLM attempts (max 3) |
| `validation_errors` | `List[str]` | Accumulated Pydantic error strings for diagnostics |

---

### Pydantic Validation & Retry Loop

Strict output validation is enforced by **Pydantic v2**. The LLM is given the
`RoleBenchmark` schema as a function-calling / JSON-mode contract via
`with_structured_output()`.

**Retry strategy:**

1. The extract node calls `chain.invoke()`.
2. If LangChain raises a `ValidationError` (malformed or schema-mismatch output), the
   error is caught, `error_count` is incremented, and the graph loops back to the
   extract node.
3. After **3 consecutive failures** the router raises a `ValueError` which the FastAPI
   layer converts to a `502 Bad Gateway` response.
4. On any successful extraction the graph proceeds directly to the embed node.

This loop guarantees that only schema-valid data ever reaches the database.

---

### pgvector Embedding Setup

After a successful extraction the `embed` node creates a **semantic fingerprint** of the
role by concatenating all extracted skill and tool strings into a single corpus sentence:

```python
corpus = ", ".join(must_have_skills + nice_to_have_skills + required_tools)
vector = embedder.embed_query(corpus)   # → List[float], len = 1536
```

The vector is stored in the `role_benchmarks` PostgreSQL table in a `Vector(1536)` column
backed by the **pgvector** extension. This enables efficient ANN (approximate nearest-
neighbour) queries for role similarity and candidate–job gap analysis.

**Model:** `text-embedding-3-small` (OpenAI) — 1536 dimensions.

## Observability

### Prometheus Metrics

Latency for the end-to-end LLM extraction (including any retries) is tracked using a
**Prometheus `Histogram`** defined at module level in `app/api/v1/benchmarks.py`.

| Metric name | Type | Labels | Description |
|---|---|---|---|
| `benchmark_extraction_latency_seconds` | Histogram | _(none)_ | Wall-clock time for the full `run_benchmark_pipeline()` call |

The metric is observed using the context-manager form so that both successful calls and
calls that ultimately raise (after retries are exhausted) are recorded:

```python
with EXTRACTION_LATENCY.time():
    final_state = run_benchmark_pipeline(raw_text=body.job_text)
```

Expose the metrics endpoint by mounting `make_asgi_app()` from `prometheus_client` in
`main.py`:

```python
from prometheus_client import make_asgi_app
app.mount("/metrics", make_asgi_app())
```

### Langfuse Tracing

Every individual LLM invocation inside the extract node is traced with **Langfuse**
via the `langfuse.callback.CallbackHandler`.

A **new handler instance** is created per attempt so that each retry appears as a
distinct, properly-labelled observation in the Langfuse dashboard:

```python
langfuse_handler = CallbackHandler(
    trace_name="role_benchmark_extraction",
    metadata={"attempt": attempt, "max_retries": MAX_RETRIES},
)
chain.invoke({"raw_text": ...}, config={"callbacks": [langfuse_handler]})
langfuse_handler.flush()   # guaranteed flush in the `finally` block
```

Each trace captures:

| Signal | Detail |
|---|---|
| **Prompt inputs** | The full system prompt + formatted human message |
| **LLM outputs** | The raw JSON / function-call response from GPT-4o-mini |
| **Latency** | Per-LLM-call duration (not just end-to-end) |
| **Attempt metadata** | `attempt` and `max_retries` tags on every span |
| **Retry correlation** | Multiple spans for the same request share the same `trace_name` |

**Required environment variables:**

```
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_SECRET_KEY=sk-lf-...
LANGFUSE_HOST=https://cloud.langfuse.com   # or your self-hosted URL
```

## Environment Variables

All configuration is read from environment variables.  Copy `.env.example` to `.env`
and fill in real values.  **Never commit `.env` to version control.**

| Variable | Required | Default | Description |
|---|---|---|---|
| `DATABASE_URL` | **Yes** | _(none — app will not start if missing)_ | SQLAlchemy connection string |
| `OPENAI_API_KEY` | **Yes** | _(none)_ | OpenAI API key for LLM + embeddings |
| `DEFAULT_LLM_MODEL` | No | `gpt-4o-mini` | Chat model for structured extraction |
| `DEFAULT_EMBEDDING_MODEL` | No | `text-embedding-3-small` | Embedding model (1536-dim) |
| `LANGFUSE_PUBLIC_KEY` | No | _(none)_ | Langfuse public key for tracing |
| `LANGFUSE_SECRET_KEY` | No | _(none)_ | Langfuse secret key for tracing |
| `LANGFUSE_HOST` | No | `https://cloud.langfuse.com` | Langfuse host URL |

> [!WARNING]
> `DATABASE_URL` has **no fallback default**. The application raises a `ValueError`
> at startup if it is not set, rather than silently connecting to an embedded
> credential. This is intentional.

## API Reference

### `POST /benchmarks/analyze`

Runs the full extraction + embedding pipeline, persists the result, and returns
the structured benchmark.

> [!IMPORTANT]
> The returned benchmark is an **AI-generated suggestion** that requires human
> review before being used in any hiring decision or candidate evaluation.
> The response always includes `"review_required": true`.

**Request:**

```json
{
  "job_text": "<raw unstructured job description, min 50 chars>"
}
```

**Success `201 Created`:**

```json
{
  "id": 42,
  "review_required": true,
  "must_have_skills": ["Distributed systems design", "API design principles"],
  "nice_to_have_skills": ["GraphQL experience"],
  "required_tools": ["Python", "FastAPI", "Docker", "PostgreSQL"],
  "experience_patterns": {
    "minimum_years": 3,
    "level": "Senior"
  },
  "common_responsibilities": [
    "Design and implement RESTful APIs",
    "Conduct code reviews and mentor junior engineers",
    "Collaborate with product and data teams"
  ]
}
```

**Error Responses:**

| Code | Cause |
|---|---|
| `422 Unprocessable Entity` | `job_text` shorter than 50 characters |
| `502 Bad Gateway` | LLM extraction failed after 3 retries |
| `500 Internal Server Error` | Database commit failed or unexpected runtime error |
